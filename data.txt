In Node.js, you should generally prefer asynchronous (async) operations over synchronous (sync) operations, especially in server applications.

Why async is usually preferred

Node.js runs JavaScript on a single main thread with an event loop. When you use an async operation, Node can continue handling other requests while waiting for I/O (file access, database queries, API calls, etc.).

Async example
const fs = require('fs');

fs.readFile('data.txt', 'utf8', (err, data) => {
  console.log(data);
});

console.log('Server can do other work');

Output:

Server can do other work
(file contents later)

The program doesn't stop while the file is being read.

What happens with sync operations

Synchronous operations block the event loop until they finish.

Sync example
const fs = require('fs');

const data = fs.readFileSync('data.txt', 'utf8');
console.log(data);

console.log('This waits until file reading is complete');

Output:

(file contents)
This waits until file reading is complete

Nothing else can be processed during readFileSync().

Real-world impact

Imagine your API receives 1,000 requests.

Async
app.get('/user', async (req, res) => {
  const user = await db.getUser(req.query.id);
  res.json(user);
});

While waiting for the database, Node can handle other requests.

Sync
app.get('/user', (req, res) => {
  const user = db.getUserSync(req.query.id);
  res.json(user);
});

Every request blocks the server until the database call completes, reducing throughput dramatically.

When sync can be acceptable
1. Startup initialization

Reading configuration files once during application startup:

const config = fs.readFileSync('config.json', 'utf8');

This happens before the server starts accepting requests.

2. Small scripts / CLI tools

For quick scripts:

const data = fs.readFileSync('input.txt', 'utf8');

Simplicity may be more important than performance.

3. Build tools and automation

Scripts run by developers or CI pipelines often use sync APIs because they are easier to write and performance is less critical.

Async patterns in modern Node.js

Today, the preferred style is usually:

const fs = require('fs/promises');

async function loadFile() {
  const data = await fs.readFile('data.txt', 'utf8');
  return data;
}

Benefits:

Non-blocking
Cleaner than callbacks
Easier error handling with try/catch
try {
  const data = await fs.readFile('data.txt', 'utf8');
  console.log(data);
} catch (err) {
  console.error(err);
}
Quick rule of thumb
Situation	Prefer
Web server/API	Async
Database queries	Async
External API calls	Async
File operations during requests	Async
App startup configuration	Sync or Async
One-off scripts	Sync or Async (based on simplicity)
High-concurrency applications	Async
Bottom line

For most Node.js applications—especially APIs, web servers, and services—use async operations because they don't block the event loop and allow Node.js to handle many requests efficiently. Use sync operations mainly for startup code, simple scripts, or situations where blocking is acceptable.

Streams are one of the most important features in Node.js. They allow you to process data piece by piece (chunks) instead of loading everything into memory at once.

Why Streams?

Imagine you have a 5 GB video file.

Without streams
const fs = require('fs');

const data = fs.readFileSync('movie.mp4');

Node tries to load the entire 5 GB file into memory before doing anything with it.

Problems:

High memory usage
Slow startup
Can crash if memory is insufficient
With streams
const fs = require('fs');

const stream = fs.createReadStream('movie.mp4');

stream.on('data', (chunk) => {
  console.log(`Received ${chunk.length} bytes`);
});

The file is read in small chunks (e.g., 64 KB at a time), making it much more memory-efficient.

Real-Life Analogy

Think of drinking water:

❌ Not a stream:

Fill a giant tank with 100 liters.
Then start drinking.

✅ Stream:

Drink directly from a flowing tap.

Streams let you consume data as it arrives.

Types of Streams in Node.js
1. Readable Stream

Used to read data.

Examples:

Reading files
Reading HTTP request bodies
Reading data from a database/network
const fs = require('fs');

const readable = fs.createReadStream('data.txt');

readable.on('data', (chunk) => {
  console.log(chunk.toString());
});
2. Writable Stream

Used to write data.

const fs = require('fs');

const writable = fs.createWriteStream('output.txt');

writable.write('Hello ');
writable.write('World');
writable.end();
3. Duplex Stream

Can read and write.

Examples:

TCP sockets
WebSockets
socket.write('Hello');
socket.on('data', (data) => {
  console.log(data.toString());
});
4. Transform Stream

Reads data, modifies it, and writes the transformed result.

Examples:

Compression
Encryption
Data formatting
const { Transform } = require('stream');

const upperCase = new Transform({
  transform(chunk, encoding, callback) {
    callback(null, chunk.toString().toUpperCase());
  }
});
Events in Streams
data

Triggered when a chunk arrives.

stream.on('data', (chunk) => {
  console.log(chunk);
});
end

Triggered when reading is complete.

stream.on('end', () => {
  console.log('Finished reading');
});
error

Triggered on errors.

stream.on('error', (err) => {
  console.error(err);
});
Piping Streams

One of the most powerful stream features is pipe().

Instead of manually reading and writing chunks:

readable.on('data', (chunk) => {
  writable.write(chunk);
});

Use:

const fs = require('fs');

const readStream = fs.createReadStream('input.txt');
const writeStream = fs.createWriteStream('output.txt');

readStream.pipe(writeStream);

Node automatically:

Reads chunks
Writes chunks
Handles flow control
Example: Copy a Large File
const fs = require('fs');

fs.createReadStream('large.mp4')
  .pipe(fs.createWriteStream('copy.mp4'));

This is much more memory-efficient than:

const data = fs.readFileSync('large.mp4');
fs.writeFileSync('copy.mp4', data);
Transform Stream Example

Convert file contents to uppercase:

const fs = require('fs');
const { Transform } = require('stream');

const upperCase = new Transform({
  transform(chunk, enc, cb) {
    cb(null, chunk.toString().toUpperCase());
  }
});

fs.createReadStream('input.txt')
  .pipe(upperCase)
  .pipe(fs.createWriteStream('output.txt'));

Flow:

input.txt
    ↓
Read Stream
    ↓
Transform Stream
    ↓
Write Stream
    ↓
output.txt
Streams in HTTP Servers

When serving large files:

const http = require('http');
const fs = require('fs');

http.createServer((req, res) => {
  const stream = fs.createReadStream('video.mp4');

  stream.pipe(res);
}).listen(3000);

Benefits:

Doesn't load the whole video into memory
Starts sending data immediately
Handles large files efficiently
Backpressure

Suppose:

Read stream produces data very fast.
Write stream consumes data slowly.

Without control, memory would fill up.

Node streams automatically manage this using backpressure.

readable.pipe(writable);

When the destination is slow:

Node pauses reading.
Resumes when the destination catches up.

This is one reason pipe() is preferred.

Interview Definition

A stream in Node.js is an object that enables reading, writing, or transforming data continuously in small chunks rather than loading the entire dataset into memory. Streams improve performance, reduce memory usage, and are commonly used for file processing, network communication, and handling large amounts of data efficiently.

Meaning

D:\NODE JS\data.txt

Check your code

You probably have something like:

Node resolves 'data.txt' relative to the current working directory (D:\NODE JS), so it's looking for:

D:\NODE JS\data.txt
Fix 1: Create the file

Create a file named:

data.txt

inside:

D:\NODE JS\

Example:

If your file is inside another folder:

D:\NODE JS

    └── data.txt

Then:

const stream = fs.createReadStream('./files/data.txt');
Fix 3: Handle stream errors

Always add an error listener when working with streams:

const fs = require('fs');

const stream = fs.createReadStream('data.txt');

stream.on('data', (chunk) => {
  console.log(chunk.toString());
});

stream.on('error', (err) => {
  console.error('File error:', err.message);
});

Without an error handler, Node crashes when the stream emits an error.

Better approach using __dirname

This avoids path issues:

const fs = require('fs');
const path = require('path');

const filePath = path.join(__dirname, 'data.txt');

const stream = fs.createReadStream(filePath);

This always looks for data.txt in the same directory as app.js.
